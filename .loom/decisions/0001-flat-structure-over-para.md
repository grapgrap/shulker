# 폴더 분류(PARA) 대신 플랫 구조와 그래프를 사용한다

## 맥락

shulker가 다루는 문서의 종류가 매우 다양하다 -- 기술 아이디어, TIL, 프로젝트 기록, 미팅, 원온원, 슬랙 대화, 기술적 문제, 잡담 등. PARA(Projects/Areas/Resources/Archive)를 조직 구조로 검토했으나, 대부분의 문서가 카테고리 경계를 넘는다. 미팅이 프로젝트 미팅일 수도 있고 Area의 정기 미팅일 수도 있으며, TIL이 프로젝트에서 나올 수도 있다. 어떤 단일 축으로도 깔끔하게 분류되지 않는다.

## 결정

폴더 계층 없이 플랫 구조를 사용하고, aeira 위키링크 그래프가 조직을 담당한다.

- 모든 문서는 단일 디렉토리에 존재한다
- 문서 간 관계는 위키링크로 표현한다
- aeira가 그래프를 구축하고 탐색을 제공한다
- digest 스킬이 사후에 관계를 엮고 정제한다

### 제약 조건

- shulker 문서의 주된 소비자는 클로드다: 파일시스템의 눈으로 보는 탐색성보다 그래프 기반 검색이 중요하다
- 문서 유형이 다양하고 횡단적이다: 어떤 단일 분류 축도 모든 문서를 깔끔하게 나누지 못한다

### 전제 조건

- aeira가 설치되어 있고 위키링크 그래프를 구축할 수 있다: 그래프가 유일한 조직 메커니즘이므로 aeira는 필수 인프라다

## 대안

### PARA를 디렉토리 구조로 사용

- 설명: Projects/Areas/Resources/Archive 폴더로 문서를 분류한다
- 트레이드오프: 대부분의 문서가 경계를 넘으며, 폴더에 넣는 순간 하나의 축에 고정되고 나머지 관계를 잃는다

### 문서 형식별 디렉토리 (notes/, meetings/ 등)

- 설명: 캡처 형식을 기준으로 폴더를 나눈다
- 트레이드오프: 형식은 내용의 본질이 아니다. 미팅에서 나온 기술 아이디어는 meeting인가 note인가

## 영향

- [[document]] - document의 조직 방식이 폴더 계층 없는 플랫 구조로 결정된다
