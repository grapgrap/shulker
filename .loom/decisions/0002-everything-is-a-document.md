# 사람, 날짜, 주제 모두 document로 표현한다

## 맥락

shulker의 문서에는 사람(동료), 날짜(시점), 주제(프로젝트, 기술 영역)와의 관계가 필요하다. 초기에는 프론트매터 메타데이터(`date`, `people`, `source`)로 이를 표현하려 했으나, 사람을 주제와 동일하게 추상화할 수 있다는 관찰이 나왔다. 날짜 역시 파일시스템 타임스탬프와 중복되며, 내용이 가리키는 시점은 문서로 표현할 수 있다.

## 결정

사람, 날짜, 주제를 모두 독립된 document로 취급하고, 관계를 위키링크로 표현한다.

- `[[김철수]]` -- 사람 document
- `[[2026-04-17]]` -- 날짜 document
- `[[auth-module]]` -- 주제 document
- 프론트매터를 사용하지 않는다
- 관계 표현 메커니즘이 위키링크 하나로 통일된다

### 제약 조건

- 관계 표현 메커니즘이 단일해야 한다: 프론트매터와 위키링크 두 가지가 공존하면 어디에 무엇을 넣을지 매번 판단해야 한다

### 전제 조건

- aeira 그래프가 위키링크를 기반으로 동작한다: 모든 관계가 위키링크이면 aeira가 전수를 탐색할 수 있다
- 날짜 document의 이웃 탐색으로 시간 범위 쿼리를 대체할 수 있다: `aeira graph neighbors "2026-04-17"`이 해당 날짜의 모든 관련 문서를 반환한다

## 대안

### 프론트매터 메타데이터로 사람/날짜 표현

- 설명: `date`, `people`, `source` 필드를 프론트매터에 둔다
- 트레이드오프: 위키링크와 프론트매터라는 두 가지 관계 표현 메커니즘이 공존하여 복잡도가 증가한다. 어떤 정보를 어디에 넣을지 기준이 모호해진다

### 날짜만 프론트매터, 나머지는 위키링크

- 설명: 시간 범위 쿼리를 위해 `date` 필드만 유지한다
- 트레이드오프: 단일 메커니즘 원칙이 깨진다. 날짜 document로 동일한 쿼리가 가능하다

## 영향

- [[document]] - document의 데이터 모델에서 프론트매터가 제거되고, 위키링크가 유일한 관계 표현 수단이 된다
